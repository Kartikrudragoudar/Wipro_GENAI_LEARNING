Create a full-stack project named orbitguard using a LOOP Engineering architecture.

Product name:
OrbitGuard - Space Debris & Satellite Collision Risk Dashboard

Product concept:
Build a SpaceTech dashboard that visualizes satellites and space debris, estimates close-approach collision risk, generates risk alerts, and helps users monitor orbital safety through an operational dashboard.

This project should feel like a real SpaceTech monitoring product, not a generic space landing page.

Core LOOP workflow:
Ingest → Normalize → Propagate → Detect → Score → Alert → Review → Improve

Each loop stage must be visible in the UI and represented in the backend architecture.

Main user journey:
1. User opens the OrbitGuard dashboard.
2. User loads sample satellite/debris data or imports TLE data.
3. System normalizes orbital records.
4. System estimates simplified orbital positions.
5. System detects possible close approaches.
6. System calculates collision risk scores.
7. System generates alerts and a risk timeline.
8. User reviews object details and risk reports.
9. System improves monitoring through filters, saved watchlists, and historical risk trends.

Important scope rule:
Build an MVP with simplified orbital calculations first.
Do not claim high-precision orbital prediction.
Clearly label results as simulated or educational estimates unless using a validated orbital propagation library.

Frontend tech stack:
- Next.js with TypeScript

- Recharts for risk charts and timelines

Backend tech stack:

- Optional orbital library if available: sgp4

- Future support for Gemini, OpenAI, or Claude

LOOP stages:

1. Ingest Loop
- User loads sample satellite/debris catalog or imports TLE data.
- System validates the dataset.
- System creates an orbital monitoring session.
- UI shows ingest status, object count, and dataset metadata.

Supported MVP data sources:
- bundled sample TLE dataset
- uploaded TLE text file
- uploaded CSV object catalog

Sample CSV format:
object_id,name,type,operator,country,altitude_km,inclination_deg,raan_deg,mean_motion,epoch
25544,ISS,satellite,NASA/International,International,420,51.6,0,15.5,2026-06-28
DEB001,Debris Fragment 001,debris,Unknown,Unknown,780,98.4,12,14.2,2026-06-28

Object types:
- satellite
- debris
- rocket_body
- unknown

2. Normalize Loop
- Backend normalizes orbital records into a consistent object model.
- Parse TLE lines if provided.
- Clean missing or invalid fields.
- Convert numeric values.
- Tag object category.
- Calculate data quality score.
- Flag records that need review.

UI should show:
- total objects
- valid objects
- invalid objects
- satellites count
- debris count
- rocket bodies count
- data quality score

3. Propagate Loop
- Backend estimates simplified orbital positions.
- If sgp4 is available, use it for TLE propagation.
- If not, use simplified circular orbit approximation for MVP.
- Generate object positions for a time window.
- Time window examples:
  - now
  - next 1 hour
  - next 6 hours
  - next 24 hours
- Store position samples for visualization and risk checks.

Important:
Label simplified calculations clearly as educational estimates.

UI should show:
- current estimated object positions
- orbit paths
- selected time window
- propagation method used: sgp4 or simplified

4. Detect Loop
- Backend detects possible close approaches.
- Compare object positions within the selected time window.
- Calculate estimated distance between objects.
- Flag pairs below configurable thresholds.

Threshold examples:
- high attention: under 5 km
- medium attention: under 15 km
- low attention: under 50 km

UI should show:
- close approach list
- object pair
- estimated closest distance
- estimated time of closest approach
- threshold level

5. Score Loop
- Backend calculates a collision risk score.
- Risk score should combine:
  - closest distance
  - object type
  - relative speed estimate if available
  - data quality
  - time to closest approach
- Output risk levels:
  - low
  - medium
  - high
  - critical
  - needs_review

Risk score formula can be simplified for MVP but must be deterministic and explainable.

UI should show:
- risk score
- risk level badge
- reason for risk score
- confidence/data quality indicator

6. Alert Loop
- Backend generates risk alerts.
- Alerts should include:
  - alert_id
  - severity
  - object_pair
  - closest_distance_km
  - time_to_closest_approach
  - explanation
  - recommended_review_action
  - status

Alert statuses:
- new
- acknowledged
- monitoring
- dismissed

UI should show:
- active alerts panel
- severity filters
- alert timeline
- acknowledge/dismiss actions

7. Review Loop
- User can inspect object details, risk events, and alert history.
- User can add objects to a watchlist.
- User can filter by type, country, operator, altitude band, risk level, and time window.
- User can export a risk report.

UI should show:
- object catalog table
- object detail drawer
- watchlist
- risk event timeline
- export report button

8. Improve Loop
- System helps improve future monitoring.
- Track acknowledged alerts and watchlisted objects.
- Show historical risk trend for the current session.
- Mock AI provider generates an operations-style summary:
  - highest-risk objects
  - recurring risk zones
  - recommended monitoring priorities
  - data quality warnings
