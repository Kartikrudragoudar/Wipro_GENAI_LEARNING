Create a full-stack project named self-correction-code-assistant using a Loop Engineering architecture.

Product concept:
Build a VS Code-inspired AI developer tool that helps developers fix broken code through an iterative correction loop.

This is not a generic chatbot. It is a loop-based developer assistant where every fix goes through a clear cycle:
1. User provides broken code.
2. System analyzes the code and issue.
3. Assistant generates a fix.
4. User provides test output, error output, or feedback.
5. Assistant self-corrects the previous fix.
6. UI compares attempts.
7. The system records what changed and why.

Core Loop Engineering Model:
Use this loop as the main product workflow:

Input Loop:
- User enters code, language, error message, and optional context.
- System validates the input.
- System creates an analysis session.

Analysis Loop:
- Backend analyzes the code.
- AI provider identifies bug summary, root cause, risks, and likely fix strategy.
- UI shows the reasoning in a structured assistant panel.

Correction Loop:
- Backend generates fixed code.
- UI shows original code, fixed code, explanation, confidence score, and suggested tests.
- User can compare original vs fixed code in a diff view.

Validation Loop:
- User pastes test output, runtime error, compiler error, or feedback.
- Backend treats this as validation feedback.
- System decides whether the previous fix succeeded, failed, or needs improvement.

Self-Correction Loop:
- Backend uses original code, previous fixed code, feedback, and attempt history.
- AI provider creates an improved fix.
- UI adds a new correction attempt.
- Each attempt must show:
  - attempt number
  - timestamp
  - status
  - bug summary
  - root cause
  - fixed code
  - explanation
  - confidence score
  - what changed from previous attempt

Learning Loop:
- The app stores correction history in local state for MVP.
- Each session should preserve the sequence of attempts.
- Future architecture should allow saving sessions to a database.

Main user workflow:
1. User opens the app.
2. User selects a sample bug or pastes code.
3. User selects language: Python, JavaScript, TypeScript, Java, or C++.
4. User enters error message or issue description.
5. User clicks Analyze.
6. Backend starts the Analysis Loop.
7. Assistant returns structured analysis and first fix.
8. User reviews fixed code and diff.
9. User pastes test output or feedback.
10. User clicks Self-correct.
11. Backend starts the Self-Correction Loop.
12. UI adds the new attempt to correction history.
13. User can switch between attempts and compare changes.

Loop engine requirements:
Create a backend module named loop_engine.py.

It should coordinate the loop:
- validate_input()
- run_analysis_loop()
- run_correction_loop()
- run_validation_loop()
- run_self_correction_loop()
- build_loop_trace()

The loop engine should not directly depend on a specific AI model.
It should use an AIProvider interface.

AI provider requirements:
Create an AIProvider interface with:
- analyze_code(request)
- generate_fix(analysis)
- self_correct(request)
- summarize_attempt_change(previous_code, new_code)

Create MockAIProvider first.

MockAIProvider should return realistic results for:
- Python typo errors
- JavaScript undefined variables
- TypeScript type mismatch
- missing imports
- simple logic bugs

Important MVP rule:
Do not execute user-submitted code in the first version.
Validation feedback should come from user-pasted test output or error logs.
Design the backend so a sandboxed code runner can be added later.
